# Правила работы над проектом

## Описание проекта

- **Проект:** `Косарь` — RC-переделка аккумуляторной Greenworks 48V в управляемую тяговую платформу на мотор-колёсах от 10" гироскутера.
- **Цель текущего этапа:** прошить родную плату гироскутера, добиться медленного и безопасного дистанционного управления, затем перенести ходовую на Greenworks.
- **Основной пользователь:** Искандер; формат работы — практичный инженерный коучинг по прошивке, RC, железу, safety gates и документации.
- **Контекст:** донор — 10" гироскутер с платой предположительно `GD32F103RCT6`; передатчик — RadioMaster Boxer ELRS; есть ESP32/Arduino и навыки пайки.
- **Жёсткие ограничения:** нож Greenworks не участвует в Stage 1; Greenworks 48V не подключается к hoverboard board; e-stop/failsafe обязательны до теста на земле; upstream firmware не вендорится без license review.
- **Source of truth:** текущие инженерные документы лежат в `docs/`; агентная карта и журналы — в `agent_docs/`; долгоживущий персональный контекст дублируется кратко в Obsidian.

## Принципы работы агента

### 1. Сначала понять границы

- Явно классифицировать задачу: simple / complex / release-critical.
- Для задач с прошивкой, силовым питанием, RC, ножом, внешними интеграциями или публикацией на GitHub использовать gates: Product/Scope, Architect, Domain/Protocol/UX, Coder, QA/Test, Reviewer/Security, Release/GitOps.
- `UNKNOWN` не считать `PASS`.
- Не фейкать поддержку протоколов, железа, DNS/API/OAuth/SEO: если нужна ручная проверка или инфраструктура, фиксировать отдельным блоком.

### 2. Минимум необходимого

- Делать только то, что требуется для текущего этапа.
- Не добавлять автономию, нож, RTK, ROS, лишние абстракции или runtime-зависимости в Stage 1.
- Не копировать внешние root `AGENTS.md`, `.mcp.json`, hooks, CI workflows или скрипты без аудита и адаптации.
- Upstream firmware держать во временном checkout или `firmware/upstream/`, который игнорируется git.

### 3. Точечные изменения

- Сохранять существующие документы в `docs/`; `agent_docs/` добавляет процессную карту, а не заменяет stage runbooks.
- Не переписывать unrelated материалы и не удалять чужие изменения.
- Если важное решение меняется, добавлять новую запись в `agent_docs/adr/`, а не править старую историю задним числом.

### 4. Работа от проверяемой цели

Для каждой инженерной задачи фиксировать:

- что меняется;
- как проверить;
- что остаётся ручным/железным gate;
- какие риски блокируют ground test.

## Автономность

- Работать самостоятельно и доводить задачу до конца.
- Не спрашивать про рутинные решения; задавать один точный вопрос только если без него высокий риск ошибочного действия.
- Перед необратимыми действиями с железом, прошивкой, силовым питанием или публикацией в GitHub явно фиксировать risk gate.

## Project-Local Skills

- Canonical source: `.agents/skills/<name>/SKILL.md`.
- Platform mirrors, если понадобятся: `.claude/skills/<name>`, `.codex/skills/<name>`, `.cursor/skills/<name>` как symlink на `../../.agents/skills/<name>`.
- Новые reusable workflows оформлять как skills, не как slash-command файлы.
- Manifest всегда называется `SKILL.md`.

## При начале работы

- Читать `README.md`, затем `agent_docs/index.md`.
- Для stage-1 прошивки и RC первым делом читать:
  - `docs/01-stage-1-rc-bringup.md`;
  - `docs/02-firmware-eferu.md`;
  - `docs/03-rc-elrs-decision.md`;
  - `docs/04-safety-and-test-matrix.md`.
- Для контекста прошлых итераций читать релевантные записи из `agent_docs/development-history/`.
- Для важных решений сверяться с `agent_docs/adr/`.

## Универсальные правила

- Документы держать краткими, ссылаться на canonical файлы вместо копирования больших блоков.
- Любая работа с `.env` и секретами запрещена без явной просьбы.
- Для shell-поиска использовать `rg`/`find`; не сканировать секретные файлы.
- Для железа и прошивки сначала documentation/review, потом wiring/flash/test.

## Чек-лист перед завершением задачи

- Требования выполнены, diff прослеживается к запросу.
- Обновлена документация в `docs/` или `agent_docs/`, если задача нетривиальная.
- Для важного решения добавлена запись в `agent_docs/adr/`.
- Для нетривиальной итерации добавлена запись в `agent_docs/development-history/`.
- Проверки выполнены или явно указано, почему не выполнялись.
