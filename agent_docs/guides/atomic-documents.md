# Атомарные документы

Правило для событийных документов: одно событие — один файл.

## Где применять

Применять для append-only коллекций, где новые записи независимы друг от друга:

- `agent_docs/development-history/` — итерации разработки;
- `agent_docs/adr/` — значимые решения;
- `agent_docs/reports/` — отчёты задач, если появятся;
- `agent_docs/audits/` — аудиты и ревью, если появятся;
- `agent_docs/investigations/` — расследования багов и research notes, если появятся.

## Где не применять

Не дробить документы текущего состояния только ради атомарности:

- `AGENTS.md`
- `CLAUDE.md`
- `README.md`
- `agent_docs/architecture.md`
- `agent_docs/glossary.md`
- `agent_docs/guides/*.md`

Такие документы остаются canonical state: их читают как актуальное состояние, а не как журнал событий.

## Имена файлов

Формат:

```text
YYYY-MM-DD-HHMM-short-title.md
```

Пример:

```text
2026-06-16-1249-atomic-event-documents.md
```

Правила:

- время локальное для проекта/сессии;
- `short-title` писать латиницей в kebab-case;
- не использовать ручную сквозную нумерацию;
- не создавать общий индекс, который нужно обновлять при каждой записи;
- collection README описывает правила, но не обязан перечислять все записи.

## Чтение контекста

Фиксированного числа записей для чтения нет. Агент сам выбирает глубину по задаче.

Ориентиры:

- риск и обратимость изменения;
- затронутые файлы и подсистемы;
- связанные ADR;
- результаты поиска по ключевым словам, путям и названию задачи;
- необходимость восстановить причины прошлого решения.

## Архивация

Атомарные событийные записи не ротировать и не архивировать по возрасту. Дата в имени файла уже задаёт порядок, а старые записи не создают конфликтов при добавлении новых.
